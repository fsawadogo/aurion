# AWS Account Setup — for a fresh account

One-time setup. Maps to `AURION-CODING-WORKFLOW.md` §11 (the autonomous
loop uses `AWS_PROFILE=aurion-dev`; never grant the loop a prod profile).

Targets a solo CTO bringing up the dev environment. Production hardening
(IAM Identity Center, federated roles) is tracked separately as a
pre-Phase-5 task — don't try to do it on day one.

---

## Step 1 — Install / verify AWS CLI v2

```bash
aws --version
# Expect: aws-cli/2.x.x  (NOT 1.x — modern features missing in v1)
```

macOS install (if missing):

```bash
curl "https://awscli.amazonaws.com/AWSCLIV2.pkg" -o "/tmp/AWSCLIV2.pkg"
sudo installer -pkg /tmp/AWSCLIV2.pkg -target /
aws --version
```

`brew install awscli` also works but Homebrew sometimes lags major
versions; the pkg installer above is the path Amazon documents.

---

## Step 2 — Lock down the root account (5 minutes, do this NOW)

Root credentials should never be used after this step except for billing
changes or account closure.

1. Sign in at https://console.aws.amazon.com/ with the root email.
2. Top-right account menu → **Security Credentials**.
3. Multi-factor authentication → **Assign MFA device** → Virtual MFA
   (use 1Password, Authy, Google Authenticator, or iCloud Keychain).
4. Sign out. From now on, sign in as the IAM user from Step 3 — never
   as root.

---

## Step 3 — Create the daily-driver IAM user

For solo-dev / pilot use, an IAM user with `AdministratorAccess` + MFA
is acceptable. Production-GA (Tier B in the deployment plan) will
migrate to IAM Identity Center.

In the AWS Console (still signed in as root for this one step):

1. **IAM** → Users → Create user.
2. User name: `faical-cto` (or any human name — not "admin").
3. Check **Provide access to AWS Management Console**.
   - Console password → autogenerated.
   - Require password reset on next sign-in → on (you'll reset to
     something the password manager generates).
4. Permissions → **Attach policies directly** → `AdministratorAccess`.
5. Create user.
6. Download the .csv with the temporary password OR copy the password
   to a password manager.
7. Sign out of root.
8. Sign in as the new IAM user. Reset the password.
9. **Security credentials** tab on the user → Multi-factor authentication
   → Assign MFA device (virtual MFA, same flow as root). Required.
10. **Security credentials** → Create access key → use case:
    **Command Line Interface (CLI)** → confirm.
11. Save the Access Key ID + Secret Access Key in the password manager.
    The secret is shown exactly once.

## Step 4 — Configure the local CLI profile

The profile MUST be named `aurion-dev` — the autonomous-loop MCP config
in `.claude/mcp/mcp.json` references this name.

```bash
aws configure --profile aurion-dev
# AWS Access Key ID:     <paste>
# AWS Secret Access Key: <paste>
# Default region name:   ca-central-1
# Default output format: json
```

Make it the default for this shell so you don't fat-finger flags:

```bash
echo 'export AWS_PROFILE=aurion-dev' >> ~/.zshrc
exec zsh   # reload
```

## Step 5 — Verify

```bash
aws sts get-caller-identity
```

Expected output:

```json
{
  "UserId": "AIDAxxxxxxxxxxxx",
  "Account": "123456789012",
  "Arn": "arn:aws:iam::123456789012:user/faical-cto"
}
```

Write down the **Account ID** — it's part of every state bucket and
secret ARN you'll touch.

## Step 6 — Billing alarm (5 minutes — do this before the bootstrap)

Fresh accounts get surprise bills. Set up a guardrail before any
`terraform apply`.

1. Console → **Billing and Cost Management** → Billing preferences.
2. Check **Receive AWS Free Tier alerts** + **Receive CloudWatch
   billing alerts**.
3. Save preferences.

Then in **CloudWatch (us-east-1 — billing metrics only exist in
us-east-1, even for ca-central-1 accounts)**:

1. Alarms → Create alarm → Select metric → Billing → Total Estimated
   Charge → USD.
2. Conditions: Static, Greater than, **$20**. (Bump after a month of
   live pilot traffic — typical pilot run-rate is $80–$150/month.)
3. Notification → Create new SNS topic → name `aurion-billing-alerts`
   → your email.
4. Confirm the subscription email when it arrives.
5. Create alarm.

## Step 7 — Run the bootstrap

```bash
cd ~/Documents/GitHub/Aurion/infrastructure/bootstrap

terraform init
terraform plan     # eyeball: ~7 resources, all KMS/S3/DynamoDB
terraform apply    # ~30s — confirm `yes`
```

After apply, emit the backend config files:

```bash
terraform output -raw backend_dev  > ../backends/dev.s3.tfbackend
terraform output -raw backend_prod > ../backends/prod.s3.tfbackend
```

Switch the main module to remote backend:

```bash
cd ..
terraform init -reconfigure -backend-config=backends/dev.s3.tfbackend
# Terraform offers to migrate state → confirm `yes`
terraform plan -var-file=environments/dev.tfvars
```

Last command should report **No changes** (if you've previously applied
dev locally) OR list new dev resources (if this is your first dev apply).

If new resources, proceed:

```bash
terraform apply -var-file=environments/dev.tfvars
```

Stop here. **Phase 2 (TLS + DNS + WAF) is the next code change.**

## Common gotchas

- **`Unable to locate credentials`** — `AWS_PROFILE` env var isn't set,
  or the profile name in `~/.aws/credentials` is wrong. Run
  `aws configure list-profiles` to see what's registered.
- **`The security token included in the request is expired`** — if you
  ever rotate to SSO/Identity Center, run `aws sso login --profile aurion-dev`
  to refresh. With long-lived IAM user keys, this doesn't happen.
- **`AccessDeniedException` from terraform apply** — the IAM user
  needs `AdministratorAccess`. If you scoped it down, you'll be
  fighting permission whack-a-mole. Stay with Admin for dev, harden
  for prod via IAM Identity Center + scoped roles.
- **Wrong region** — every `aws` and `terraform` command should be
  hitting `ca-central-1`. `aws configure get region --profile aurion-dev`
  to confirm.

## When you graduate to IAM Identity Center (before Phase 5)

The TODO for production:

1. Enable AWS Organizations + IAM Identity Center on the account.
2. Create permission sets: `AurionAdmin`, `AurionDeploy`,
   `AurionReadOnly`, `AurionIncidentResponse`.
3. Migrate `aurion-dev` and create `aurion-prod` as SSO profiles.
4. Delete the long-lived IAM user access keys.
5. Update `.claude/mcp/mcp.json` to reference SSO sessions.

That work is part of Phase 5 (operational readiness) in the deployment
plan, not Phase 1. Don't optimize prematurely.
